# Personal Operating Model Command

Create a comprehensive Personal Operating Model (POM) that defines how you create value, your strengths/weaknesses, failure patterns, decision system, and execution playbook.

## Purpose

This command invokes the `personal-operating-model` skill to interactively create your operating manual covering:
- **Core Loop**: How you transform inputs into outputs
- **Strengths**: Energy-positive capabilities with evidence
- **Unique Edge**: Your asymmetric advantages
- **Weaknesses**: Structural limitations to route around
- **Pitfalls**: Repeating failure patterns with interrupt rules
- **Decision System**: How you should decide (defaults, rules, metrics)
- **Operating Constraints**: Your guardrails and non-negotiables
- **Energy & Rhythm**: Peak windows and burnout indicators
- **Environment Design**: Friction management and amplifiers
- **Execution Playbook**: Your default shipping process
- **Experiments**: System improvements to test

**When to use**: Creating your first POM, major career transitions, annual POM updates, or when patterns have shifted significantly.

## Usage

```bash
/planning-strategy:personal-operating-model
```

**No parameters needed** - the skill guides you through an interactive interview.

## How It Works

This command invokes the `personal-operating-model` skill, which:

1. **Fetches context** from Notion:
   - Year Plan (current focus, OKRs, strategic themes)
   - Golden Rules (working principles)
   - Founder Wisdom (patterns to emulate)
   - Previous POM (if exists)
   - Optional: Recent Quarterly Logs (pattern finding)

2. **Guides you through 11 sections**:
   1. Core Loop (Input ‚Üí Transformation ‚Üí Output)
   2. Strengths (energy-positive capabilities)
   3. Unique Strong Points (asymmetric edge)
   4. Weaknesses (structural limitations)
   5. Pitfalls (repeating failure patterns)
   6. Decision System (how you decide)
   7. Operating Constraints (guardrails)
   8. Energy & Rhythm (cognitive weather map)
   9. Environment Design (friction management)
   10. Execution Playbook (how you ship)
   11. Experiments (system improvements)

3. **Asks 3-5 deep questions per section** (not fill-in-the-blank)

4. **Synthesizes answers** into actionable insights

5. **Reviews for consistency** and asks follow-ups

6. **Saves to Notion** POM page

## Time Commitment

**Total**: 60-90 minutes of focused time

**Per section**: 5-8 minutes
- 3-5 interview questions
- Synthesis and confirmation
- Move to next section

**Don't rush this**. Depth > speed.

## What You'll Get

**Comprehensive POM Document** covering:

### 1. Core Loop ‚Äî How You Create Value
- Inputs you transform best (and should avoid)
- Transformation mechanism (your signature move)
- Outputs that compound
- One-line core loop: "I transform X into Y by Z, producing W"

### 2. Strengths ‚Äî Energy-Positive Capabilities
For each strength (3-5 total):
- What you do better than most
- Why it works (mechanism)
- Evidence (2-3 examples with outcomes)
- Contexts where it shines vs weakens
- How to exploit it more

### 3. Unique Strong Points ‚Äî Asymmetric Edge
- Your rare combination (A + B + C)
- Unfair advantage this creates
- Best contexts and timing
- Problems that fit you
- When you win vs lose

### 4. Weaknesses ‚Äî Structural Limitations
For each weakness (3-6 total):
- What consistently underperforms
- Trigger conditions
- Cost when ignored
- Best mitigation strategy
- Environmental fix
- "Good enough" standard

Categorized as:
- Skill gaps (can train)
- Energy drains (should avoid)
- Blind spots (must design around)

### 5. Pitfalls ‚Äî Repeating Failure Patterns
For each pitfall (3-5 total):
- Pattern (what you do)
- Early warning signals
- Typical rationalization
- Damage if unchecked
- **Interrupt rule** (hard rule, not advice)
- Recovery protocol

### 6. Decision System ‚Äî How You Should Decide
**Decision Defaults**:
- Speed (fast/medium/slow)
- Confidence threshold (60%/80%/95%)
- What you tend to overdo

**Decision Rules by Type**:
- Opportunities (new projects/partnerships)
- Commitments (long-term contracts/hires)
- Daily Execution (what to do today)

**Evaluation Function**:
- Leading indicators
- Lagging indicators
- Non-negotiable indicators

### 7. Operating Constraints ‚Äî Guardrails
- Non-negotiables (what you won't do)
- Capacity constraints (max parallel projects/meetings)
- Default "No" criteria

### 8. Energy & Rhythm ‚Äî Cognitive Weather Map
- Peak windows (deep work, execution, social)
- Low-energy protocol (safe vs danger tasks)
- Burnout indicators (3 warning signs)
- Recovery protocol (minimum viable)

### 9. Environment Design ‚Äî Make Right Behavior Easier
- Add friction to (stop bad habits)
- Remove friction from (ship more)
- Tools that amplify you
- People archetypes you work best with
- Structures that keep you honest

### 10. Execution Playbook ‚Äî How You Ship
- Default project phases (5 steps)
- Definition of "Done"
- Quality bar (minimum)
- Shipping cadence
- Anti-stall mechanisms

### 11. Experiments ‚Äî Improve the Model
For each experiment (1-3 total):
- Hypothesis
- Change (what you will do)
- Duration (2-week cycles recommended)
- Success metric
- Failure metric
- Review date

**Saved to**: https://www.notion.so/Your-POM-Page

## Tips for Best Results

### 1. Block Focused Time
- **90 minutes uninterrupted**
- Turn off notifications
- This is strategic work, not admin

### 2. Be Brutally Honest
- This is NOT a resume or LinkedIn profile
- Acknowledge real weaknesses (they're constraints, not flaws)
- Name actual pitfalls (failures you've repeated 3+ times)
- No aspirational BS ("I want to be more organized" ‚ùå)

### 3. Use Concrete Examples
- ‚úÖ "I consistently over-architect. Last 3 projects: spent 2 weeks on frameworks before shipping anything. Cost: missed launch windows."
- ‚ùå "I sometimes overthink things."

### 4. Push for Evidence
- For strengths: "I did X, which resulted in Y outcome with Z metric"
- For weaknesses: "This failed when... (specific instance)"
- For pitfalls: "3 examples: project A (2023), project B (2024), project C (2025)"

### 5. Design, Don't Wish
- Weaknesses are constraints to route around, not problems to fix
- Pitfalls need interrupt rules ("If X, then stop and do Y"), not motivation
- Environment design beats willpower

### 6. Reference Your Context
- The skill pulls from Year Plan, Golden Rules, Founder Wisdom
- Build on that context
- Point out where POM conflicts with Year Plan (resolve it)

### 7. Make It Actionable
After reading your POM, you should know:
- What work to seek vs avoid
- What decisions to make fast vs slow
- What warning signs to watch for
- What experiments to run next

## Common Pitfalls When Creating POM

**Avoid These**:

1. **Resume Mode**: Writing aspirational strengths instead of honest assessment
   - ‚ùå "I'm a visionary leader who inspires teams"
   - ‚úÖ "I'm good at extracting core mechanisms from complex systems (evidence: X, Y, Z)"

2. **Vague Weaknesses**: Generic statements without mitigation
   - ‚ùå "I'm not great at details"
   - ‚úÖ "I skip operational details. Mitigation: use checklist for client onboarding. Partner with detail-oriented PM for delivery."

3. **Motivational Pitfalls**: Treating failure patterns as motivation problems
   - ‚ùå "I need to be more disciplined about shipping"
   - ‚úÖ "Pitfall: Over-architecting. Interrupt rule: If outline >3 pages, ship outline first. Recovery: Cut to core, ship v1."

4. **Generic Experiments**: Testing abstractions instead of specifics
   - ‚ùå "Be more focused"
   - ‚úÖ "Experiment: Max 2 parallel projects for 30 days. Success: ship both. Failure: start project #3."

5. **Ignoring Context**: Creating POM in isolation from Year Plan/Golden Rules
   - Connect strengths to Year Plan OKRs
   - Align constraints with Golden Rules
   - Reconcile conflicts explicitly
